# SOC L1 Fundamentals: Systems Analysis & Alert Triage

**Objective:** สรุปองค์ความรู้พื้นฐานสำหรับ Security Operations Center (SOC) Analyst Level 1 โดยเน้นความเข้าใจในกระบวนการเกิดช่องโหว่ของระบบ (Vulnerabilities), การตั้งค่าที่ผิดพลาด (Misconfigurations) และวงจรการจัดการแจ้งเตือนภัยคุกคาม (Alert Triage Lifecycle)

---

## Part 1: Systems as Attack Vectors (เจาะลึกการโจมตีผ่านระบบ)

### 1.1 Why Systems Matter (ทำไมระบบถึงตกเป็นเป้าหมาย)
การโจมตีระบบโดยตรงมักสร้าง **"ผลกระทบที่รุนแรงและเป็นวงกว้าง"** มากกว่าการโจมตีผู้ใช้ทั่วไป (Social Engineering) ตัวอย่างเช่น หากผู้โจมตีสามารถยึดครอง Mail Server ได้ จะสามารถเข้าถึงข้อมูลของพนักงานทุกคนในองค์กรได้ทันที

### 1.2 Vulnerability Management (การจัดการช่องโหว่ของซอฟต์แวร์)
ช่องโหว่คือข้อผิดพลาดในระดับโค้ดของซอฟต์แวร์ที่ผู้พัฒนาสร้างขึ้น
* **CVE (Common Vulnerabilities and Exposures):** รหัสอ้างอิงมาตรฐานสำหรับช่องโหว่ที่ถูกค้นพบและเปิดเผยแล้ว (เช่น `CVE-2023-29357`)
* **Zero-day:** ช่องโหว่ร้ายแรงที่ผู้โจมตีค้นพบก่อนผู้พัฒนาซอฟต์แวร์ ทำให้ยังไม่มีตัวแก้ (Patch) ออกมาป้องกัน
* **The Response:** การรับมือที่ดีที่สุดคือการอัปเดต Patch ทันที แต่ระหว่างรอ SOC ต้องใช้มาตรการชั่วคราว เช่น IP Restriction หรือ WAF/IPS

### 1.3 Misconfigurations (ภัยเงียบจากการตั้งค่าที่ผิดพลาด)
แตกต่างจากบั๊กของโปรแกรม เพราะเกิดจากความมักง่ายหรือความผิดพลาดของมนุษย์ในการตั้งค่าระบบ (เช่น ใช้รหัสผ่านเริ่มต้น "123456" หรือเปิดพอร์ตทิ้งไว้) การอัปเดตซอฟต์แวร์ไม่สามารถแก้ปัญหานี้ได้
* **Proactive Defense:** การป้องกันเชิงรุกต้องใช้การทำ **Penetration Testing**, **Vulnerability Scans** และ **Configuration Audits**
* **Remediation:** แก้ไขที่ต้นเหตุโดยใช้ Secure Password Policy และ Security Training for IT

## Part 2: SOC Alert Triage Lifecycle (การจัดการและการสืบสวนภัยคุกคาม)

### 2.1 From Events to Alerts
* **Event:** ทุกเหตุการณ์ที่เกิดขึ้นในระบบ (เช่น การล็อกอิน, การโหลดไฟล์)
* **Log:** บันทึกของเหตุการณ์นั้นๆ 
* **Alert:** การแจ้งเตือนที่ถูกกรองและสร้างขึ้นโดยระบบ SIEM หรือ EDR เมื่อพบ Log ที่มีพฤติกรรมเข้าข่ายน่าสงสัย เพื่อให้ SOC Analyst เข้ามาตรวจสอบ

### 2.2 Alert Prioritisation (การจัดลำดับความสำคัญ)
เมื่อมี Alert จำนวนมากเข้ามาใน Dashboard กฎการเลือกเคสมาจัดการมีดังนี้:
1. **Sort by Severity:** เลือกระดับความรุนแรงสูงสุดก่อนเสมอ (`Critical` > `High` > `Medium` > `Low`)
2. **Sort by Time:** หากระดับความรุนแรงเท่ากัน ให้เลือก Alert ที่ **เก่าที่สุด (Oldest)** ก่อน เพราะหมายถึงผู้โจมตีอาจอยู่ในระบบมานานกว่า

### 2.3 The Triage Process (ขั้นตอนการสืบสวน)
การทำงานของ SOC Analyst กับ Alert แต่ละตัว จะมี Workflow มาตรฐานดังนี้:
1. **Initial Actions:**
   * ตรวจสอบ Alert ที่อยู่ในสถานะ `New`
   * Assign เคสให้ตัวเองเป็นผู้รับผิดชอบ
   * เปลี่ยนสถานะเป็น `In Progress`
2. **Investigation (การสืบสวนเชิงลึก):**
   * วิเคราะห์ข้อมูลจาก Logs เพื่อหาความผิดปกติ (ใครคือผู้ตกเป็นเป้าหมาย, เกิดการกระทำอะไรขึ้น, มีพฤติกรรมน่าสงสัยก่อน-หลังเหตุการณ์หรือไม่)
3. **Final Actions & Verdict:**
   * สรุปผลการสืบสวนว่าเป็นภัยคุกคามจริง (**True Positive**) หรือเป็นเพียงการแจ้งเตือนผิดพลาดจากการใช้งานปกติ (**False Positive**)
   * เขียนคอมเมนต์อธิบายเหตุผลการตัดสินใจให้ชัดเจน
   * เปลี่ยนสถานะเป็น `Closed` เพื่อจบการทำงาน
